# 案例采集规范

本文档定义三类案例的分类标准、案例格式规范和写入流程。

---

## 1. 三类案例分类标准

### 类型一：通用 ARM 适配问题（`G-` 系列）

**写入文件**：`migration-cases/01-generic-arm-migration.md`

**判断标准**：根本原因是 **x86 与 ARM 指令集 / ABI / 内存模型 / 系统环境的架构差异**，换一个项目到 ARM 上仍然会碰到同类问题。

**属于此类的典型问题：**
- x86 专属编译标志（`-mf16c`、`-mavx2`、`-mpopcnt`、`-mssse3`）在 ARM 上不支持
- `char` 符号性差异（x86 默认有符号，ARM 默认无符号）
- x86 SIMD intrinsics（SSE/AVX）在 ARM 上找不到（`immintrin.h`、`emmintrin.h`）
- x86 专属汇编指令（`bts`、`btr`、`bt`、`rdtsc`、`cpuid`）→ ARM 汇编替代
- ARM 弱内存序导致的 `atomic_thread_fence` 缺失问题
- 预编译库格式不匹配（x86 库在 ARM 上 `file in wrong format`）
- `_GLIBCXX_USE_CXX11_ABI` 新旧 ABI 不兼容
- 硬编码绝对路径（`/home/xxx/`）跨机器不可移植
- `PAGE_SIZE` 宏不可移植 → `sysconf(_SC_PAGESIZE)` 替代
- 头文件隐式包含差异（ARM GCC 不隐式包含 `<functional>`、`<atomic>` 等）
- 构建文件 `export_incs` / `includes` 传递性缺失
- `__builtin_ia32_*` 等 GCC x86 builtins 在 ARM 上未定义
- 信号处理中的析构顺序问题（x86 隐式强序，ARM 弱序暴露问题）
- ARM 环境全局对象析构顺序不同导致的崩溃

**不属于此类（要区分）：**
- 工具链版本升级导致的问题 → 归入 V- 系列
- 项目特有依赖组合问题 → 归入 P- 系列

---

### 类型二：版本兼容性问题（`V-` 系列）

**写入文件**：`migration-cases/02-version-compatibility.md`

**判断标准**：根本原因是**新旧版本之间的不兼容变更**，与 ARM/x86 架构本身无关——即使在同一架构上升级工具版本也会遇到同类问题。在 ARM 迁移中"被触发"，是因为 ARM 系统环境自带的工具版本更新。

**属于此类的典型问题：**
- 解析器生成工具版本升级导致 API 变更（旧指令废弃）
- Python 2 → Python 3 语法不兼容（`raise Exception, msg` 旧语法）
- GCC 版本升级后更严格的警告（`-Wreturn-type`、`-Wsign-compare`、`-Wstrict-aliasing`）
- Protobuf 版本不兼容（旧版 `.pb.h` 与新版 protoc 头文件冲突）
- Boost 旧版本不识别 ARM 编译器版本宏
- CMake 版本升级后的语法变化

---

### 类型三：项目特有问题（`P-` 系列）

**写入文件**：`migration-cases/03-project-specific.md`

**判断标准**：根本原因是**该项目特有的业务依赖组合、配置遗漏或历史欠债**，换一个不同业务的项目不会遇到同类问题。

**属于此类的典型问题：**
- 特定内部框架的 ARM 版 BUILD 文件欠债
- 项目内特定 protoc 路径指向错误（ABI 版本选择错误）
- 项目内特定第三方库的 BUILD 文件未对齐（依赖注释掉了没恢复）
- 特定业务组件（如内部监控、限流等）ARM 版本缺失或处理逻辑不同
- 项目信号处理与特定 框架析构顺序的特有组合问题
- 业务特定的打点格式不一致
- 特定文件字段类型不一致
- 特定cmds 脚本（如架构转换辅助脚本）

---

## 2. 案例文件格式规范

每条案例的标准格式（与 `sourcecode-error-patterns.md` 保持一致）：

```markdown
### <ID>：<简洁标题（说明问题本质，不含项目名）>

**错误现象：**
```
<实际编译报错信息，完整粘贴，可多行，已脱敏>
```

**根因分析：**
<一段话说明为什么 x86 正常但 ARM 出错，或为什么版本升级导致此问题。
不超过 5 行，突出关键差异。>

**修复方法：**

修改前：
```<语言>
<改动前代码，已脱敏，路径使用通用占位符>
```

修改后：
```<语言>
<改动后代码>
```

**适用场景：**
<一行说明：什么情况下会遇到这个问题，适用范围>

**验证方式：**
```bash
<验证命令，确认修复已生效>
```

**diff patch：**
完整 diff patch 已保存到报告目录：`<迁移工作目录>/reports/arm_migration_YYYYMMDD_HHMMSS.patch`

相关改动涉及文件：
- `<涉及文件路径 1>`
- `<涉及文件路径 2>`（如有）

**扫描规则：**（仅 G- 系列案例填写，V-/P- 系列留空）
```
<用户确认后由 AI 填入以下内容，或写“未加入扫描规则” / “DevKit 已覆盖”>
规则 ID：SCAN-G<NN>-01
匹配文件类型：<如 .cpp .h .cc .hpp>
匹配模式：<正则表达式描述>
已写入 arm-scan-rules.json：<是 / 否>
```

---
```

### 格式要求

1. **ID 格式**：`G-NN`、`V-NN`、`P-NN`（两位数字，不足补零），从文件现有最大 ID + 1 开始
8. **扫描规则**（仅 G- 系列）：
   - 入库后必须调用 `AskQuestion` 询问用户是否将匹配规则加入 `arm-scan-rules.json`
   - 用户确认后填入扫描规则字段：规则 ID、匹配文件类型、匹配模式和写入状态
   - 用户选择「否」或「DevKit 已覆盖」时，填写对应原因
   - DevKit 已完整覆盖的类型（汇编/intrinsics/SIMD 头文件等）不加入扫描规则
2. **标题**：说明问题本质，**不包含**项目名、仓库名、机器名等
3. **错误现象**：尽量保留完整编译报错，帮助后续匹配。多行错误全部保留
4. **根因分析**：说明“为什么”，而不只是描述“发生了什么”
5. **代码块语言**：`cpp`、`python`、`bash`、`cmake`、`makefile`、`bison` 等
6. **diff patch**：**必填**，记录报告目录中的 patch 文件路径 + 改动涉及的文件列表。注意：
   - **diff 内容保存到报告目录中（`<迁移工作目录>/reports/`），不内嵌到案例库文件中**
   - diff 内容**无需脱敏**，保留真实路径和内容
   - 案例中只需写明 patch 文件的绝对路径 + 改动涉及的文件列表
   - 无实际 patch 文件时（仅有对话上下文），填写 `无，请提供分支/目录信息`
7. **脱敏要求**（仅针对案例库文件本身，不包含报告目录下的 diff 文件）：
   - 仓库地址 → `ssh://git@<your-host>/<repo>.git` 或 `https://<your-package-repo>/...`
   - 内网 IP → `<internal-host>`
   - 用户 home 目录 → `/home/<username>/`
   - 业务名/服务名 → `<service-name>`、`<project-name>`
   - 具体 sha256/commit hash 保留格式但替换值 → `<sha256-hash>` / `<commit-hash>`

## 3. 写入流程

### 3.1 追加新案例

**始终追加到文件末尾**（`---` 分隔符之前），不修改已有案例：

```bash
# 检查当前最大 ID
grep "^### G-" migration-cases/01-generic-arm-migration.md | tail -1
# 确定新 ID = 最大 ID + 1
```

### 3.2 更新文件头部索引

每个案例库文件顶部有案例索引，新增案例后需要更新：

```markdown
案例分组：
- **G-01 系列**：xxx
- **G-02 系列**：xxx（新增）
```

### 3.3 同时更新 sourcecode-error-patterns.md（可选）

对于**高频通用错误**（预计多个项目都会遇到），同时更新 `sourcecode-error-patterns.md` 中对应系列。

## 4. 分类决策树

```
遇到一个迁移改动，如何分类？

是否是工具链/库版本升级导致的 API 变化？
├─ YES → V- 系列（版本兼容性）
│         示例：Bison 3.x 废弃指令、Python 3 语法变化、GCC 更严格警告
└─ NO ↓

改动是否只在这一个特定项目中发生、换个项目不会有？
├─ YES → P- 系列（项目特有）
│         示例：特定 RPC 框架 BUILD 欠债、特定业务协议类型不一致
└─ NO ↓

是否根本原因是 x86/ARM 架构差异（指令集/ABI/内存模型/系统环境）？
├─ YES → G- 系列（通用 ARM 适配）
│         示例：x86 汇编→ARM 汇编、内存序、char 符号性、ABI 不兼容
└─ 不确定 → 归入 G- 系列并在根因中注明不确定性
```

## 5. 案例质量检查清单

写入前自检：

- [ ] 错误现象是完整的编译报错或运行时错误信息
- [ ] 根因分析说明了“为什么 x86 正常但 ARM 出错”的机制
- [ ] 代码示例包含“修改前”和“修改后”
- [ ] **`diff patch` 字段已填写**，包含报告目录中的 patch 文件绝对路径和涉及文件列表
- [ ] 所有仓库地址、IP、用户名、业务名已脱敏（案例库文件本身，不包括报告目录下的 diff 文件）
- [ ] 适用场景一行说明清楚，帮助后续检索
- [ ] 验证方式可以实际执行（不是伪代码）
- [ ] ID 未与现有案例冲突
- [ ] 文件头部索引已更新
